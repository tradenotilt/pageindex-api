# Дизайн: поддержка `file_name` в `POST /upload`

## Контекст

Сейчас `POST /upload` принимает только файл и использует `file.filename` как источник имени для валидации, определения типа, сохранения временного файла и отображения в задачах/реестре.

## Цель

Добавить необязательный аргумент `file_name` в multipart/form-data, чтобы клиент мог переопределять имя файла для обработки без ломки текущих интеграций.

## Принятое решение

- Добавить параметр `file_name: str | None = Form(None)` в endpoint `POST /upload`.
- Если `file_name` передан и не пустой после `strip()`, использовать его как итоговое имя файла.
- Если `file_name` не передан, использовать текущее поведение с fallback на `file.filename`.
- Если оба значения пустые, использовать имя `document`.

## Влияние на систему

- Валидация расширения и размера остается прежней, но выполняется по итоговому имени файла.
- Определение `file_type` выполняется по итоговому имени файла.
- Временный файл сохраняется с итоговым именем.
- В `TaskManager` и реестр попадает итоговое имя, поэтому дополнительные изменения контракта не требуются.

## Обработка ошибок

- Пустой `file_name` не считается ошибкой и игнорируется.
- Если в итоговом имени нет поддерживаемого расширения, endpoint возвращает ту же ошибку валидации, что и раньше.

## Документация

- Обновить `README.md`: описание потока, примеры использования и поведение нового параметра `file_name`.

## Проверка

- Загрузка только с `file` продолжает работать без изменений.
- Загрузка с `file` и `file_name` использует имя из `file_name` в задачах, реестре и типе файла.
