# 🗂 PageIndex API

Human-like Document AI Indexer — твой личный архивариус на базе ИИ.

**Powered by PageIndex**

⚠️ **Дисклеймер:** Этот проект является независимой реализацией API для библиотеки PageIndex и не аффилирован с официальной командой разработчиков.

## 🚀 Быстрый старт

```bash
git clone https://github.com/tradenotilt/pageindex-api.git .
cd pageindex-api

cp .env.example .env
# Отредактируйте .env и добавьте свои API ключи!
```

## 🐳 Установка и запуск (Docker)

После того как вы настроили файл `.env`, выполните следующие шаги в терминале:

### Шаг 1: Сборка образа

На этом этапе Docker прочитает инструкции из Dockerfile и подготовит изолированную среду, установив все необходимые зависимости.

```bash
docker compose build
```

### Шаг 2: Запуск сервера

Теперь запускаем наш API-сервер в фоновом режиме. Флаг `-d` (detached) позволит сервису работать даже после того, как вы закроете терминал.

```bash
docker compose up -d
```

### Проверка статуса

Чтобы убедиться, что всё запустилось успешно, введите:

```bash
docker ps
```

Если в колонке `STATUS` вы видите `Up`, значит, ваш персональный ИИ-архивариус готов к работе!

### Почему мы делаем это в два этапа?

Разделение команд позволяет вам контролировать процесс:

- **Build** — проверяет, что весь код корректен и библиотеки установились без ошибок
- **Up** — просто активирует уже готовый, проверенный образ

## ✅ Как проверить работу?

После выполнения команд откройте браузер:

```
http://IP_ВАШЕГО_СЕРВЕРА:8000/docs
```

Вы попадете в интерактивную документацию Swagger. Не забудьте авторизоваться, используя ваш `APP_API_KEY` из файла `.env`, чтобы получить доступ к загрузке документов.

⚠️ **ВАЖНО:** Никогда не выкладывайте свой `.env` файл в публичный доступ! Файл `.env` добавлен в `.gitignore`.

## 📖 Как это работает

1. Пользователь загружает файл через [`POST /upload`](app.py:362)
2. Файл сохраняется в рабочую папку и передаётся в очередь задач
3. [`TaskManager`](task_manager.py:1) вызывает [`FileHandler.process_file()`](file_handler.py:87)
4. Для PDF используется каскадная обработка:
   - сначала пытается [`PageIndex`](file_handler.py:111) с автоматическим обогащением таблицами
   - затем Markdown-мост с распознаванием таблиц
   - затем улучшенный резервный режим с распознаванием таблиц через camelot
5. Для PDF с изображениями выполняется Vision OCR через GPT-4o Vision
6. Создается [`document_index`](file_handler.py:393) с автоматической генерацией keywords через GPT-4o-mini
7. Результат сохраняется в JSON внутри папки [`data/`](data/)
8. Поиск по документу выполняется через [`POST /search`](app.py:311):
   - сначала модель выбирает релевантные узлы
   - затем система собирает контекст
   - если структура слабая, используется fallback по релевантным фрагментам
9. Все эндпоинты защищены API-ключом через заголовок `X-API-Key`

## 🛠 Фичи

- **Умный индекс:** сохранение структуры документа (PageIndex)
- **Распознавание таблиц:** автоматическое извлечение таблиц из PDF с помощью camelot
- **Vision OCR:** автоматическое описание графиков и изображений через GPT-4o
- **Document Index:** автоматическое создание индекса документа с keywords через LLM
- **Полная приватность:** все данные хранятся локально в папке `data/`
- **Async Task Manager:** фоновая обработка тяжелых файлов
- **Поддержка форматов:** PDF, DOC, DOCX, Markdown
- **Гибридный парсинг:** PageIndex + распознавание таблиц для максимальной точности

### Document Index с автоматической генерацией keywords

Система автоматически создает индекс документа в начале каждого JSON-файла с информацией о:

- **Секции документа:** заголовки, диапазоны страниц, количество чанков
- **Таблицы:** названия, страницы, описания
- **Изображения:** названия, страницы, описания
- **Keywords:** автоматически генерируются через выбранную модель для каждого элемента

**Преимущества Document Index:**
- **Улучшенный контекст для LLM:** модель сразу видит структуру документа
- **Оптимизированный поиск:** можно быстро фильтровать по секциям и ключевым словам
- **Лучшая навигация:** пользователь видит структуру и может переходить к нужным разделам

**Выбор модели для keywords:**

По умолчанию используется `gpt-4o` для генерации keywords. Вы можете изменить модель в `.env`:

```env
# Для максимального качества keywords (рекомендуется)
KEYWORDS_MODEL=gpt-4o

# Для экономии токенов (быстрее, но менее точно)
KEYWORDS_MODEL=gpt-4o-mini

# Для бюджетного варианта
KEYWORDS_MODEL=gpt-4o-mini
```

**Выбор модели для Vision OCR:**

По умолчанию используется `gpt-4o` для описания изображений. Вы можете изменить модель в `.env`:

```env
# Для максимального качества описания (рекомендуется)
VISION_MODEL=gpt-4o

# Для экономии токенов
VISION_MODEL=gpt-4o-mini
```

### Vision OCR для графиков и изображений

Система автоматически извлекает изображения из PDF и описывает их через GPT-4o Vision:

- **Автоматическое распознавание:** графики, диаграммы, схемы
- **Контекстное описание:** каждый график описывается в контексте вашего запроса
- **Интеграция в поиск:** описания добавляются в структуру документа
- **Ограничение:** до 5 изображений на документ для экономии токенов

**Пример использования:**
```bash
# Загрузите PDF с графиками через /upload
# При поиске система автоматически учтет описания графиков
```

### Загрузка документа через `/upload`

Эндпоинт принимает `multipart/form-data` со следующими полями:

- `file` — обязательный файл для обработки
- `file_name` — необязательное имя файла для обработки; если передано, оно заменяет исходное имя файла из загрузки

Если `file_name` не передан, API использует исходное имя из `file.filename`. Именно итоговое имя применяется для определения расширения, валидации и записи в задачи/реестр.

**Пример запроса:**

```bash
curl -X POST "http://IP_ВАШЕГО_СЕРВЕРА:8000/upload" \
  -H "X-API-Key: your_private_api_key" \
  -F "file=@./report.tmp" \
  -F "file_name=report.pdf"
```

Такой запрос полезен, если исходный локальный файл имеет временное или служебное имя, а для обработки нужно явно передать корректное имя с расширением.

## 🔧 Настройка

### Создание файла `.env`

Скопируйте пример и заполните своими значениями:

```bash
cp .env.example .env
```

Обязательные параметры в `.env`:

```env
CHATGPT_API_KEY=your_openai_key
APP_API_KEY=your_private_api_key
```

⚠️ **Важно:** `APP_API_KEY` — это ваш личный пароль для доступа к вашему серверу, а не ключ OpenAI. Используйте надежный пароль для защиты вашего API.

Дополнительные параметры (опционально):

```env
DATA_DIR=./data
RESULTS_DIR=./results
MAX_FILE_SIZE=52428800

# Модель PageIndex (по умолчанию: gpt-4o-2024-11-20)
PAGEINDEX_MODEL=gpt-4o-2024-11-20

# Количество страниц для проверки оглавления (по умолчанию: 50)
PAGEINDEX_TOC_CHECK_PAGES=50

# Модель для генерации keywords в document_index (по умолчанию: gpt-4o)
KEYWORDS_MODEL=gpt-4o

# Модель для Vision OCR (по умолчанию: gpt-4o)
VISION_MODEL=gpt-4o

# Принудительное использование улучшенного парсинга с распознаванием таблиц
# Установите в true, если PageIndex плохо справляется с документами
FORCE_ENHANCED_PARSING=false
```

### Улучшенный парсинг таблиц

Если PageIndex плохо справляется с вашими документами (не распознает таблицы, плохая структура), установите `FORCE_ENHANCED_PARSING=true` для принудительного использования улучшенного парсинга с распознаванием таблиц через camelot.

**Преимущества улучшенного парсинга:**
- Автоматическое распознавание таблиц в PDF
- Создание отдельных семантических узлов для каждой таблицы
- Сохранение структуры таблиц в читаемом формате
- Текстовые превью для лучшего понимания содержимого
